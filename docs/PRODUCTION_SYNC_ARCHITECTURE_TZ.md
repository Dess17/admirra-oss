# ТЗ на будущую production-архитектуру синхронизации

Дата: 2026-06-12

## Контекст

Текущая синхронизация Admirra работает на одном сервере и использует встроенный фоновой worker внутри backend-процесса. Это подходит для текущей стадии разработки и умеренной нагрузки, но для настоящего production с большим количеством пользователей нужно вынести синхронизацию в отдельную отказоустойчивую архитектуру.

Цель будущей доработки: синхронизация не должна теряться, дублироваться, зависать при перезапуске backend, превышать лимиты внешних API и блокировать веб-приложение.

## Что уже сделано сейчас

- Синхронизация запускается через таблицу `sync_jobs`.
- Ручной sync запускает полный период через `force_full`.
- Автоматический/фоновый sync может использовать инкрементальный период.
- На уровне одного backend-процесса есть параллельное выполнение задач.
- Добавлен атомарный захват queued job через БД.
- Добавлен общий rate-limit внутри одного процесса для запросов к внешним API.
- Статус и `last_sync_at` хранятся в `integrations` и отображаются на dashboard/projects/integrations.

## Ограничения текущей реализации

- Worker живёт внутри backend-процесса. При рестарте процесса текущие in-memory слоты теряются.
- При нескольких backend-инстансах нужен полноценный distributed lock/queue, иначе возможны гонки вокруг задач и лимитов.
- Rate-limit сейчас общий только внутри одного процесса, а не между несколькими серверами.
- Нет отдельной панели очереди синхронизации для администратора.
- Нет retry policy по типам ошибок с отдельными dead-letter задачами.
- Нет полноценного backpressure, который регулирует количество задач по API-платформам и пользователям.

## Требуемая production-архитектура

### 1. Вынести sync worker из backend

Нужно разделить:

- `web/backend` — принимает запросы, отдаёт UI/API, создаёт sync jobs.
- `sync-worker` — выполняет синхронизацию Яндекс.Директ, Яндекс.Метрика, VK Ads, Avito Ads.
- `scheduler-worker` — создаёт плановые задачи автоматической синхронизации.
- `report-worker` — генерирует отчёты, PDF/AI/Google Sheets/рассылки.

Рекомендуемый стек:

- Celery + Redis/RabbitMQ, либо RQ + Redis.
- Для production предпочтительнее RabbitMQ или managed Redis с persistence.

### 2. Очереди по типам задач

Разделить очереди:

- `sync.yandex_direct`
- `sync.yandex_metrika`
- `sync.vk_ads`
- `sync.avito_ads`
- `reports`
- `notifications`
- `maintenance`

Это позволит ограничивать тяжёлые задачи Яндекса отдельно от быстрых задач и не блокировать всю систему одним типом API.

### 3. Distributed rate-limit

Нужен rate-limit, общий для всех worker-инстансов:

- по платформе;
- по OAuth-токену/аккаунту;
- по рекламному кабинету;
- по пользователю/проекту.

Минимальный вариант:

- Redis-based token bucket;
- ключи вида `rate:yandex_direct:{account_login}`, `rate:metrika:{counter_id}`, `rate:vk:{account_id}`.

Обязательное поведение:

- при 429 задача не падает сразу, а откладывается на `Retry-After`;
- лимиты не умножаются при добавлении новых worker-инстансов;
- администратор видит, что задача задержана из-за лимита API.

### 4. Job lifecycle

У каждой задачи должны быть состояния:

- `QUEUED`
- `RUNNING`
- `RETRYING`
- `SUCCESS`
- `FAILED`
- `CANCELLED`
- `STALE`

Хранить:

- `created_at`
- `started_at`
- `finished_at`
- `next_retry_at`
- `attempt`
- `max_attempts`
- `stage`
- `progress`
- `error_code`
- `error_message`
- `payload`
- `result_summary`

### 5. Идемпотентность

Синхронизация должна быть безопасной при повторном запуске одной и той же задачи.

Требования:

- upsert статистики по уникальным ключам: integration/campaign/date;
- удаление и перезапись диапазона целей Метрики только в рамках синхронизируемого периода;
- отдельный `sync_batch_id` для группового запуска;
- защита от одновременной синхронизации одной integration.

### 6. Batch-синхронизация проекта

При ручном sync проекта нужно создавать batch:

- `sync_batches`
  - id
  - client_id
  - created_by
  - status
  - started_at
  - finished_at
  - requested_days
  - force_full

Каждая integration получает свою job, но UI показывает общий batch progress.

Пользовательский UI:

- на странице проекта: “Синхронизация проекта: 2/3 канала”;
- на странице integrations: статус по каждому каналу;
- после завершения: “Проект обновлён: сегодня, 14:35 МСК”.

### 7. Автоматическая синхронизация

Scheduler должен:

- запускаться отдельно от web;
- выбирать integrations с `auto_sync=true`;
- учитывать `sync_interval`;
- не создавать дубль, если уже есть активная задача;
- использовать incremental sync;
- учитывать паузу проекта;
- учитывать лимиты тарифа и статус интеграции.

### 8. Ручная синхронизация

Ручная синхронизация должна:

- запускать `force_full=true`;
- синхронизировать выбранный период;
- обновлять статистику, цели, направления, топ креативы, балансы;
- после завершения инвалидировать кеш dashboard/project cards;
- возвращать batch/job id для UI polling.

### 9. Наблюдаемость и админка

Добавить admin-инструменты:

- список активных sync jobs;
- список failed jobs;
- фильтр по пользователю/проекту/интеграции;
- ручной retry;
- cancel job;
- просмотр payload/result/error;
- среднее время sync по платформам;
- количество 429/401/403/500 от внешних API.

### 10. Мониторинг и алерты

Метрики:

- длина очередей;
- время ожидания в очереди;
- длительность sync job;
- количество failed/retry jobs;
- rate-limit events;
- API latency по платформам;
- DB pool usage;
- количество stale jobs.

Алерты:

- очередь sync растёт дольше N минут;
- job выполняется дольше лимита;
- 429/401/403 резко выросли;
- worker не отправляет heartbeat;
- нет успешных sync за последние N часов.

### 11. База данных

Для production:

- managed PostgreSQL HA;
- минимум 2 зоны доступности;
- ежедневные backup;
- PITR, если доступно;
- отдельная read replica при росте аналитических запросов.

Обязательно:

- миграции Alembic должны быть единственным способом изменения схемы;
- убрать ручные ALTER TABLE на production;
- все новые таблицы/индексы делать через миграции.

### 12. Deployment

Нужно разделить сервисы:

- `backend-web`
- `sync-worker`
- `scheduler-worker`
- `report-worker`
- `redis/rabbitmq`

Деплой:

- rolling deploy;
- health checks;
- graceful shutdown worker: не убивать running job мгновенно;
- retry interrupted jobs после рестарта.

### 13. Безопасность

- Все API-токены хранить в зашифрованном виде.
- Секреты хранить в secret manager, не в репозитории.
- Ограничить доступ worker к минимуму нужных переменных.
- Логи не должны содержать access token, refresh token, private key, client secret.

### 14. Приоритет внедрения

1. Вынести sync worker в отдельный процесс.
2. Добавить Redis/RabbitMQ очередь.
3. Добавить distributed lock на integration.
4. Добавить distributed rate-limit.
5. Добавить sync batch для проекта.
6. Добавить админку очереди.
7. Добавить мониторинг и алерты.
8. Перевести scheduler на отдельный worker.

## Ожидаемый результат

После реализации production-архитектуры:

- веб-приложение не блокируется синхронизацией;
- задачи не теряются при рестарте;
- одна integration не синхронизируется параллельно дважды;
- лимиты внешних API соблюдаются глобально;
- пользователь видит понятный статус проекта и каждого канала;
- администратор может диагностировать зависшие и упавшие задачи.
